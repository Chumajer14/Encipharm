1. Resumen Ejecutivo

Enci / Enci Ventas es un sistema web de gestión comercial y MiniCRM diseñado para centralizar la operación del equipo de ventas. Permite administrar clientes, registrar interacciones, controlar oportunidades, gestionar propuestas y visualizar indicadores mediante dashboards diferenciados por rol.

El MVP está construido con una arquitectura web primero: frontend React/Vite, backend FastAPI, autenticación con Firebase Auth y persistencia en Cloud Firestore. El backend actúa como capa obligatoria de autorización, validación, seguridad y escritura de datos.

Estado general: las capacidades principales del MVP están cerradas para Development. El proyecto está en etapa de preparación de UAT, validación con usuarios reales y salida controlada.

2. Qué Hace El Sistema

  • Autentica usuarios mediante Google SSO usando Firebase Auth.
  • Valida tokens Firebase en el backend antes de permitir acceso a endpoints protegidos.
  • Sincroniza usuarios autenticados en Firestore.
  • Controla acceso según rol: vendedor, supervisor y admin.
  • Permite crear, listar, buscar, filtrar, editar y eliminar lógicamente clientes.
  • Restringe a vendedores para que operen solo clientes propios.
  • Permite registrar interacciones comerciales asociadas a clientes.
  • Gestiona oportunidades comerciales mediante etapas de pipeline.
  • Permite crear propuestas vinculadas a oportunidades y calcular montos finales en backend.
  • Entrega dashboards con métricas propias para vendedores y consolidadas para supervisores/admin.
  • Expone documentación técnica y consola Swagger protegida para pruebas.
  • Incluye endpoint de readiness para validar condiciones mínimas antes de UAT/go-live.

Dependencias backend uv Instalación reproducible y ejecución de tests.
Deploy previsto Cloud Run / Vercel frontend Backend en GCP y frontend web independiente.

Arquitectura Funcional

  • El usuario entra al frontend y ejecuta login con Google.
  • Firebase Auth entrega un ID Token al frontend.
  • El frontend envía el token al backend en Authorization: Bearer <token>.
  • FastAPI valida el token con Firebase Admin.
  • El backend consulta Firestore para obtener rol, estado y permisos.
  • Las operaciones de negocio se ejecutan únicamente a través de la API.
  • Firestore no queda expuesto como escritura directa desde cliente.

Endpoints Principales

Área Endpoints
Salud y readiness GET /health, GET /readiness
Usuario actual GET /me
Autenticación POST /auth/login, POST /auth/register
Clientes GET /clientes, POST /clientes, GET /clientes/{id}, PATCH /clientes/{id}, DELETE /clientes/{id}, POST /clientes/import-csv
Flujo comercial GET/POST /interacciones, GET/POST /oportunidades, PATCH /oportunidades/{id}, GET /oportunidades/{id}/detalle, GET/POST /propuestas, PATCH /propuestas/{id}
Dashboard GET /dashboard/vendedor, GET /dashboard/supervisor
Usuarios GET /users, GET /users/{uid}, PATCH /users/{uid}, PATCH /users/{uid}/role, PATCH /users/{uid}/status

Reglas De Autorización

  • vendedor: opera clientes, interacciones, oportunidades y propuestas propias.
  • supervisor: accede a información consolidada y funciones de supervisión.
  • admin: accede a gestión completa, cambios de rol/estado e importación CSV.
  • La autorización definitiva vive en backend; el frontend solo mejora la experiencia de navegación.

8. Seguridad Y Protecciones Extra

Riesgo Control implementado
Acceso sin sesión Endpoints protegidos exigen token Bearer Firebase válido.
Usuario desactivado sigue usando token activo=false bloquea login, /me y endpoints protegidos.
Acceso a datos ajenos Control de propiedad por vendedorUid y rol.
Eliminación irreversible Clientes usan baja lógica con deletedAt.
Uso directo de Firestore desde cliente Reglas Firestore versionadas con denegación total.
Ráfagas o abuso básico Rate limit de 120 solicitudes por minuto por IP.
Payloads sobredimensionados Límite global de 1 MB por request.
CSV gigante o inválido CSV limitado a 1 MB, 1000 filas y UTF-8.
Formula injection Rechazo de textos iniciados con =, +, - o @.
CORS permisivo Configuración rechaza * en producción.
Token persistente en Swagger Uso de sessionStorage y no persistencia permanente.
Sesión expirada Evento global frontend fuerza logout y retorno a login.
Rutas incorrectas o sin permiso Páginas 404 y 403 controladas.
Click doble o errores silenciosos Controles de guardado, disabled states y manejo de errores en pantallas comerciales.

10. Observaciones

  • El sistema está en condiciones de demo ejecutiva y UAT controlado.
  • La principal brecha actual es operativa, no funcional: falta validación con ambiente y usuarios reales.
  • El backend concentra las reglas críticas, por lo que no depende de controles visuales del frontend para proteger datos.
  • El uso de filtros en memoria sobre Firestore es aceptable para MVP, pero debe evolucionar con índices y queries server-side al crecer el volumen.
  • La importación CSV está disponible en backend para admin; debe confirmarse si el cliente necesita interfaz visual para esa operación.
  • La migración histórica debe mantenerse fuera de alcance hasta recibir fuente de datos, reglas de normalización y criterios de calidad.
  • Mobile, SAP, IA, Calendar y reportes avanzados deben tratarse como Fase 2.

11. Riesgos Residuales

Riesgo Impacto Mitigación recomendada
Firebase real no validado en UAT Bloqueo de login o permisos en ambiente cliente. Crear cuentas de prueba por rol y ejecutar checklist UAT.
Consultas Firestore en memoria Riesgo de rendimiento al aumentar volumen. Migrar a queries con índices por vendedorUid, clienteId, estado y etapa.
Sin E2E automatizado Mayor riesgo de regresión visual o de navegación. Implementar Playwright con login mockeado o ambiente de prueba.
Migración histórica indefinida Datos incompletos al iniciar operación real. Solicitar fuente, diccionario, reglas de limpieza y piloto de importación.
Reglas comerciales pendientes de confirmación Ajustes tardíos en estados, permisos o aprobaciones. Validar pipeline, estados, roles y aprobaciones en reunión ejecutiva.

12. Mejoras Recomendadas

  • Agregar pruebas frontend con Vitest y React Testing Library.
  • Agregar pruebas E2E con Playwright para flujos críticos.
  • Migrar filtros Firestore a consultas optimizadas con índices.
  • Agregar unicidad de email en alta manual si el negocio lo exige.
  • Crear pantalla de importación CSV para admin si será usada por operación.
  • Agregar exportación de clientes, oportunidades y propuestas.
  • Mejorar dashboards con tendencias temporales y métricas de conversión.
  • Agregar recordatorios de seguimiento comercial.
  • Permitir configuración de etapas de pipeline si el cliente tiene proceso propio.
  • Preparar integración futura con ERP/SAP.
  • Planificar módulo mobile como Fase 2.
  • Centralizar logger frontend y reducir mensajes no necesarios en producción.
